# Architecture Review — Validation Checklist

**Definition of Done**: This checklist ensures the architecture review is complete and ready for stakeholder delivery.

## 📋 Context & Input Validation

- [ ] Architecture documentation loaded (diagrams, ADRs, design documents)
- [ ] System constraints identified (scale, performance, regulatory, budget)
- [ ] Stakeholder list confirmed (architects, tech leads, platform team)
- [ ] Review scope clearly defined (new system, refactoring, incremental improvement)

## 🔍 Analysis Completeness

- [ ] Scalability dimension reviewed (10x growth readiness, scaling strategy)
- [ ] Reliability dimension reviewed (failure modes, recovery mechanisms, SLO targets)
- [ ] Maintainability dimension reviewed (code organization, abstraction clarity, knowledge transfer)
- [ ] Operational complexity dimension reviewed (runbooks, team skill requirements, deployment process)
- [ ] Cost efficiency dimension reviewed (resource utilization, optimization opportunities)

## 📊 Findings Quality

- [ ] Each finding assigned severity level (Critical, High, Medium, Low)
- [ ] Evidence documented for all findings (metrics, examples, pattern mismatch)
- [ ] Recommendation provided for each finding (what to change, why, effort estimate)
- [ ] Trade-offs documented (between competing concerns, effort vs benefit)

## 📝 Output & Documentation

- [ ] Architecture Decision Records (ADRs) generated for key recommendations
- [ ] Action items created with assigned owners and deadlines
- [ ] Strengths and areas for improvement clearly articulated
- [ ] Priority roadmap defined (next 6-12 months)
- [ ] Review report formatted and ready for presentation

## 🔚 Final Status

- [ ] Review outcome decided (Approve / Improve / Redesign)
- [ ] Decision rationale documented in report
- [ ] Stakeholder sign-off obtained (or path to sign-off clear)
- [ ] Output document saved to versioned location
- [ ] Team alignment confirmed on next steps
